## 3. v0.5 推荐方向

**选 #1 raw_resumes retry 工具**。理由：

1. **自然延伸**：v0.4d 已搭好状态机（processing/parsed/failed），retry 是最直接的下一步
2. **零架构变化**：纯加 1 个 tool + 1 个状态转换路径，无新依赖
3. **用户感知最强**：失败的简历可直接 retry，不用重传文件（v0.4d 之前的状态）
4. **测试基线完整**：v0.4d 已有 3 测试（raw_text 保留 + call order + 不崩溃），retry 加 2-3 测试即可
5. **价值/风险比最高**：5 候选中风险最低、价值最高

## 4. v0.5 #1 任务拆解（拆 2 PR：v0.5a refactor + v0.5b retry 工具）

### v0.5a — 抽公共函数 + 恢复 v0.4d 完整测（0.5d，1 commit）

**目标**：refactor 公共函数有"先验"基线，retry 工具开发时不被"v0.4d 成功测缺失"拖后腿。

```
apps/api/app/tools/resume_parser.py:
  - 抽 _do_extract_and_link(raw_resume_id, content) 公共函数
    - parse_resume (auto_create=True 路径) 和 retry_raw_resume 都调
    - 状态机更新集中：processing → parsed/failed 都在这里
  - _handle_parse_resume 减薄成"参数校验 + dispatch + file download"
  - handler 内不再直接管 raw_resumes 状态

apps/api/tests/mcp/integration/test_resume_parser_v0_4d.py:
  - 恢复 test_v0_4d_llm_success_marks_parsed_and_links_candidate
    - 当前"不崩溃"断言不够，需验证 candidate_id 非空 + basic_info.email 真值
  - 用更稳的 mock 模式：直接 patch app.services.candidate.CandidateService
    而不是包 3 层 MagicMock 嵌套
```

**v0.5a 退出标准**：
- [ ] v0.4d 4 测试全 pass（含恢复的 LLM 成功测）
- [ ] parse_resume 行为不变（响应字段一致）
- [ ] health-check 14/14

### v0.5b — retry 工具（1d，2 commit：feat + docs）

**目标**：retry_raw_resume 工具 + 5 测试 + e2e 集成 + ship report。

```
apps/api/app/tools/resume_parser.py:
  - 新增 _handle_retry_raw_resume(raw_resume_id: str) -> dict
    - 校验 status（只接受 failed；processing/parsed 返 CONFLICT）
    - 读 raw_resumes 表（不存在返 NOT_FOUND）
    - 调 _do_extract_and_link(raw_resume_id, raw_text)
    - v0.5b 不做 force=True（语义未定，推 v0.6 单独 PR）
```
