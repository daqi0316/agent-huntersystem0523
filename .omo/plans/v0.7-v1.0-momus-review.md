async def require_admin_user_id(

## 2. v0.8 — 5 项问题

### 2.1 [P0] 数字测的"价值"问题, plan 自承"价值低"

v0.5-replan §5 原文: "14 server 并行 spawn 压测 (dev 机非 prod, 价值低)"。

v0.7-v1.0 plan §3 决策 4 选 "**安全验证** (主目标, 不做性能基准)"。

**Momus 评估**: **OK 但应明确**:
- v0.8 真实价值 = "确认 14 server 并行不死 (有 bug 早发现)"
- 不是 "找性能数字" (dev 数字 prod 无效)
- 如果 14 server 并行发现 OOM/fd 超限, 写 ADR 推 v0.8.1 修复
- **如果没发现问题**, v0.8 ship 报告就是 "14 server 并行安全, 文档而已"

**v0.8 估时 1d 包含修复吗**: plan 写 "1d 1 commit", 暗示不包含。如发现 OOM, 估时要追加 1-2d。

### 2.2 [P0] lsof/ps 命令在 Docker/CI 容器内可能不可用

plan §4 v0.8 写 "用 lsof -p <pid> | wc -l 测 fd" + "用 ps -o rss= -p <pid> 测 memory"。

**问题**:
- macOS dev 机器: lsof 在, ps 在
- Docker 容器 (CI): lsof **可能不**装 (alpine 镜像默认不装), ps 可能简化版
- **CI 跑 fail** 可能误判

**修正**:
- 用 try/except 包, fail 时记 `fd=NA` / `memory=NA` 不阻塞实验
- 或 **本地 dev 跑, CI skip** (脚本加 `if os.environ.get("CI"): skip`)

### 2.3 [P1] trial 间资源释放延迟

30 实验 = 3 场景 × 10 trial。每 trial spawn 14 server, 立即 shutdown。

**问题**:
- subprocess 退出但 fd/port/资源**未立即释放** (OS 回收有延迟)
- trial 2 立即 spawn 14 server 可能撞 port 冲突
- 30 实验**密集连发**容易 flaky

**修正**: trial 间 `asyncio.sleep(0.5)` 让资源释放, 总时长 +15s, 可接受。

### 2.4 [P1] 与 v0.4e 顺序测数字对比意义不大

plan §4 v0.8 写 "与 v0.4e 顺序测 (9208ms total, P95 898ms/server) 对比"。

**问题**:
- v0.4e 顺序 = 14 server **一个一个** spawn, 累加
- v0.8 并行 = 14 server **同时** spawn, 取 max
- 两者数字**不可比** — 顺序是 sum, 并行是 max

**修正**: 对比维度应是:
- 并行 P95 启动时间 vs 单 server 启动时间 (200-400ms)
- 如果 P95 < 2s = 没瓶颈
- v0.4e 顺序数字**不**作为 v0.8 评估基线

### 2.5 [P2] 失败率阈值未定义

plan 写 "测得 spawn 耗时 / init 耗时 / list_tools 耗时 / 失败率"。

**问题**: "失败率 > X% 算 fail" 没说。**0% 是底线** (任何 spawn fail 都要修), **5% 是可接受** (偶发端口冲突)。**没阈值 = 报告时无判断标准**。

**修正**: 报告里加 "失败率阈值 0% (任何失败都是 bug, 必查)"。

## 4. v1.0b — 4 项问题

### 4.1 [P0] JSON serialize 后缀变化, 客户端可能 break

`datetime.utcnow().isoformat()` → `"2026-06-07T10:00:00"` (无后缀)
`datetime.now(UTC).isoformat()` → `"2026-06-07T10:00:00+00:00"` (有后缀)

**问题**:
- API 返 JSON 含 timestamp 字段, 客户端解析
- v0.6a ship report §3.3 提到 "JSON serialize 输出格式变化" 是 aware/naive 差异之一
- 客户端如 `new Date("2026-06-07T10:00:00")` vs `new Date("2026-06-07T10:00:00+00:00")` 都 work (JS Date 兼容), 但**严格 ISO 8601 解析**的 client 可能 warn

**风险**: API 兼容性 — 旧客户端期望无后缀, 新后缀可能让旧解析失败

**修正**: plan 写 "改 aware 是正确方向 (DB TZ-aware 列要求), JSON 后缀差异不阻塞, 通知前端"。

### 4.2 [P1] DB 历史 naive timestamp 数据处理

**问题**:
- DB 列 `DateTime(timezone=True)` (已存在的 4 文件用 raw SQL 查询)
- 旧数据是 naive UTC
- 写 aware datetime 时, **不冲突** (DB 自动转 UTC aware)
- 读 aware 时, **naive 数据会触发 pytz warning** 或 **自动标 UTC**

**实际**: SQLAlchemy 2.0 读 naive from TZ-aware column 时, 给 naive data 加 "无时区" 标记, 不 crash。但客户端如比较 naive 和 aware datetime, **抛 TypeError**。

**修正**: plan 写 "历史 naive 数据由 SQLAlchemy 自动处理, 不需 migration, 但下游消费 aware 的代码需注意 (v0.5a 4 文件修改时同时检查下游比较)"。

### 4.3 [P1] `from datetime import UTC` Python 版本假设

plan 写 "from datetime import UTC (Python 3.11+, 项目 3.14)"。

**问题**:
- 项目 requires-python `>=3.12` (pyproject.toml)
- 3.11 才有 `UTC` 单例, 3.12 也支持
- **OK**, 但 plan 应明确 "Py 3.12+ 风格, 项目满足"

**修正**: plan 写 "Python 3.12+ 风格 (项目 requires-python>=3.12, 满足), `from datetime import UTC` 单例 import"。

### 4.4 [P2] 4 文件现有测试可能用 naive 比较

**问题**:
- 改 aware 后, 现有 test_rate_limit_test / test_privacy_test / test_csm_test 内部用 `datetime.utcnow()` 比较 timestamp
- 改 aware 后, 比较 naive vs aware 抛 TypeError

**修正**: plan 写 "实施时**同时**扫现有 4 文件测试, 把 naive 比较改成 aware aware 比较"。
