### 1.2 [P0] Celery vs RQ 未选

v0.6 draft 写 "Celery/RQ 后台任务", 没说用哪个:
- **Celery**: 重 (broker/result/worker/beat), 适合复杂任务链 + 定时任务
- **RQ**: 轻 (Redis broker 一键), 适合简单后台任务

**建议选 RQ**:
- 项目已有 Redis (Phase Z 已 ship)
- 任务单一 (parse_resume), 不需要复杂任务链
- RQ 部署简单 (rq worker -u redis-url)
- 1 个 worker 进程即可 (vs Celery worker + beat + result backend)

**RQ vs Celery 关键差异**:
- RQ: `from rq import Queue; queue.enqueue(callable, *args)`
- Celery: `app.task decorator; task.delay(*args)`
- RQ 简单但扩展性弱 (v0.6 单任务够用)

## 5. v1.0b Draft — 3 项问题

### 5.1 [P0] datetime naive vs aware 行为差异

`datetime.utcnow()` 返 tz-**naive** UTC。
`datetime.now(UTC)` 返 tz-**aware** UTC。

**差异**:
- JSON serialize: isoformat() 输出不同 ("...Z" vs "...+00:00")
- DB 存: `DateTime(timezone=True)` 列存 aware, naive 会被 pytz 警告
- 比较: naive vs aware 抛 TypeError

**v1.0b 风险**:
- 现有代码用 utcnow() 的地方, 比较时可能混用
- DB 存时 pytz 警告 (CLAUDE.md 提到的 4 个 deprecation warnings 之一)

**v1.0b 必须加 1 测**:
- 验证 timestamp 字段是 tz-aware (v0.5a LLM 测恢复模式: 直接验属性, 不 MagicMock 嵌套)
- 验 4 文件改完, 现有测试仍 pass

### 5.2 [P1] Python 3.11+ `from datetime import UTC`

Python 3.11 之前 UTC 是 `datetime.timezone.utc`。3.11+ 才有 `UTC` 单例。

项目用 **Python 3.14**, 没问题。但 import 写 `from datetime import UTC` 是 Py 3.11+ 风格。

**v1.0b 统一用 `datetime.now(datetime.UTC)`** (兼容老 Py), 或 `datetime.now(UTC)` (新风格)。**用后者** (与 Python 3.14 + 类型注解风格一致)。
